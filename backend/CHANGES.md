## Внесённые изменения

### 1. Докстринги

Добавлены краткие русские докстринги ко всем модулям, классам и функциям бэкенда:

- **database.py**: докстринги модуля и функции для управления сессией БД
- **repositories.py**: докстринги классов и методов репозиториев
- **models.py**: докстринги ORM моделей
- **schemas.py**: докстринги Pydantic-схем
- **service.py**: краткие докстринги функций бизнес-логики
- **app.py**: докстринги для эндпоинтов
- **tasks.py**: докстринги для задач Celery

Все докстринги оформлены в кратком стиле и на русском языке.

### 2. Логирование (loguru)

Интегрировано `loguru` для структурированного логирования по всему бэкенду:

- **database.py**: нет логов (уровень инфраструктуры)
- **repositories.py**: нет логов (уровень доступа к данным)
- **service.py**: INFO/DEBUG для операций с файлами, WARNING для ошибок
- **app.py**: DEBUG для вызовов эндпоинтов, INFO для загрузок, WARNING для удаления
- **tasks.py**: INFO при запуске задач, DEBUG на этапах обработки, WARNING/ERROR при проблемах

Конфигурация `loguru` в `src/__init__.py`:
- консольный вывод с цветами
- файловый вывод в `logs/app.log` с ротацией 500 МБ и хранением 10 дней
- разные форматы для консоли и файла

### 6. Оптимизация и улучшения надёжности

#### Асинхронное удаление файлов
- Заменено синхронное `Path.unlink()` на асинхронное `aiofiles.os.remove()` в `service.py`
- Исключено блокирование worker потока при удалении больших файлов

#### Валидация размера файла
- Добавлена проверка максимального размера при загрузке (10 МБ) в `create_file()`
- Возвращается `413 Request Entity Too Large` при превышении лимита
- Проверка выполняется на уровне приложения, а не только в задачах

#### Retry логика Celery задач
- Все Celery задачи (`scan_file_for_threats`, `extract_file_metadata`, `send_file_alert`) настроены на автоматическое повторение
- При возникновении исключения задача повторяется до 3 раз с интервалом 5 секунд
- Повышает надёжность обработки при временных ошибках подключения к БД или другим сервисам

### 7. Оптимизация БД

#### Индексы для производительности
- Создана миграция `0001_add_indexes.py` с тремя индексами:
  - `ix_files_created_at` на `files.created_at` для оптимизации сортировки
  - `ix_alerts_file_id` на `alerts.file_id` для быстрого поиска алертов по файлу
  - `ix_alerts_created_at` на `alerts.created_at` для сортировки алертов
- Индексы значительно улучшают производительность запросов при росте количества записей

### 8. API пагинация и аутентификация

#### Пагинация API
- Добавлены параметры `skip` и `limit` к эндпоинтам `GET /files` и `GET /alerts`
- Параметры: `skip` (default 0), `limit` (default 20, max 100)
- Ответ возвращает: `items`, `total`, `skip`, `limit`
- Позволяет клиентам эффективно загружать большие наборы данных

#### API Key аутентификация
- Проверка `X-API-Key` задаётся глобально для приложения FastAPI (все маршруты)
- API ключ читается из переменной окружения `API_KEY` с дефолтом `test-key-dev`
- Необходимо передавать заголовок `X-API-Key: {API_KEY}` с каждым запросом
- Неправильный или отсутствующий ключ возвращает `403 Forbidden`

### 9. Исправления целостности (инфраструктура и клиент)

- **Сервисный слой:** сессия БД передаётся из эндпоинтов через `Depends(get_db)`, а не через `Depends` внутри функций, вызываемых напрямую из роутов
- **Celery:** используется `async_session_maker` из `src/database.py` (единый DSN с приложением); поддержан fallback `CELERY_BROKER_URL`, если не задан `REDIS_URL`
- **Alembic:** `migrations/env.py` импортирует `DB_URL` из `src.database`, а не из `service`
- **Docker:** проброс Postgres `5433:5432`, `depends_on` для Redis у `backend` и `backend-worker`
- **Фронтенд:** разбор ответов пагинации (`items` / `total`), заголовок `X-API-Key` при загрузке файлов, скачивание через `fetch` + blob (ключ в заголовке)
- **Примеры `.env`:** согласованы с `database.py` (`POSTGRES_*`, `PGPORT`, `REDIS_URL`, `API_KEY`)
